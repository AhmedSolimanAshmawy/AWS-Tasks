# 🚀 Step 5 — Verify the Website Endpoint

After enabling Static Website Hosting, AWS automatically creates a **Website Endpoint**.

Unlike the normal S3 REST API Endpoint, the Website Endpoint understands browser requests such as:

```
GET /
```

and automatically returns:

```
index.html
```

instead of expecting an object key.

# 🚀 Step 7 — Delegate DNS Management to Route 53

Go to **Namecheap**.

Open:

```
Domain List
    ↓
Manage
    ↓
Nameservers
```

Choose:

```
Custom DNS
```

Replace the existing Name Servers with the four Name Servers generated by Route 53.

This transfers **DNS management** from Namecheap to Route 53 while **keeping the domain registration at Namecheap**.

# 🚀 Step 8 — Create an Alias Record

Now that Route 53 manages the DNS, we need to direct the domain to the S3 Website Endpoint.

Go to:

```
Hosted Zone
    ↓
Create Record
```

Configure:

- Type:

```
A
```

- Alias:

```
Enabled
```

- Target:

```
S3 Website Endpoint
```

This allows users to access:

```
bittco.site
```

instead of the long S3 website endpoint URL.

# 🌍 DNS Propagation

Immediately after changing the Name Servers, the website **did not** start working.

Instead, it continued to open the previous website for some time.

This behavior is completely normal and is commonly referred to as:

> **DNS Propagation**

DNS Propagation is a general term describing the time required for DNS changes to become visible across the Internet.

Several components may contribute to this delay:

- 🌍 Registry updates after changing the authoritative Name Servers
- 🖥️ Operating System DNS Cache
- 🌐 Browser DNS Cache
- 🔄 Recursive Resolver Cache (ISP, Google DNS, Cloudflare, etc.)

Once cached records expire according to their **TTL (Time To Live)**, recursive resolvers query the new authoritative DNS servers (Route 53), retrieve the updated records, cache them again, and the new website becomes accessible.
